# CEF 源码学习计划（从入门到深入）

适用对象：已经集成过 CEF Wrapper、能独立编译 CEF 的开发者。  
目标：从“会用 CEF”升级到“能读懂、定位、修改 CEF 核心逻辑”。

---

## 0. 总体学习节奏（建议 10 周）

- 第 1-2 周：架构和运行链路入门（先建立全局地图）
- 第 3-4 周：API 到实现的映射（会追调用栈）
- 第 5-6 周：多进程/IPC/生命周期（理解 CEF 核心差异）
- 第 7-8 周：网络、资源加载、JS 交互（常用业务能力）
- 第 9-10 周：改源码、打 patch、验证回归（进入可贡献状态）

每周建议投入：6-10 小时。  
每个阶段都要有“可运行结果 + 学习笔记 + 源码定位记录”。

---

## 1. 学习准备（第 1 周，简单）

### 1.1 建立本地源码地图

重点目录（基于你的路径）：

- `chromium/src/cef`：CEF 自身核心代码
- `chromium/src/content`：Chromium 内容层（CEF 常依赖）
- `chromium/src/chrome`：浏览器壳层（辅助理解）
- `chromium/src/components`：通用组件能力

建议做一份“目录说明表”，记录每个目录主要职责。

### 1.2 建立调试工具链

- 能稳定执行你已有的构建脚本（如 `build_cef.sh`）
- 准备调试器（`gdb` 或 `lldb`）
- 准备搜索工具（`rg`）
- 准备记录工具（一个固定笔记文件）

### 1.3 第一批必懂概念

- CEF 是 Chromium 的嵌入封装，不是独立浏览器内核
- Browser Process / Renderer Process 的职责分离
- CEF API（`include/cef_*`）与实现（`libcef/*`）的关系

阶段产出：

- 一张你自己画的“CEF 总体架构图”
- 一份“进程模型”简表（每个进程负责什么）

---

## 2. 从 API 反查实现（第 2-3 周，入门到进阶）

目标：看到一个 CEF API，能找到它最终落到哪段实现。

### 2.1 建议切入 API（由易到难）

- `CefInitialize`
- `CefBrowserHost::CreateBrowser`
- `CefClient` 回调（如 `OnBeforeBrowse`, `OnProcessMessageReceived`）

### 2.2 学习方法

对每个 API 按同一模板追踪：

1. 看头文件注释（`include/`）
2. 找实现入口（`libcef/`）
3. 继续追到 Chromium 层（`content/` 或 `components/`）
4. 记录“线程 + 进程 + 调用链”

建议输出格式（每个 API 一条）：

- API 名称
- 调用入口文件
- 关键实现文件
- 所在线程
- 涉及进程
- 常见坑点

阶段产出：

- 至少 10 个 API 的“调用链记录卡片”

---

## 3. 多进程与 IPC（第 4-5 周，核心）

目标：真正理解 CEF 与普通库的本质差异。

### 3.1 必学主题

- Browser/Renderer 进程创建与生命周期
- 进程间消息（CEF message + Chromium Mojo/IPC）
- 线程模型（UI/IO/File 等）

### 3.2 建议实践

- 写一个简单进程消息例子（Browser 发消息到 Renderer，再回传）
- 在关键回调打日志，确认消息路径与线程切换

阶段产出：

- 一张“进程与消息时序图”
- 一份“线程限制清单”（哪些 API 只能在哪个线程调用）

---

## 4. 页面加载、网络与资源处理（第 6-7 周，业务高频）

目标：掌握业务中最常见扩展点。

### 4.1 重点模块

- `CefRequestHandler`
- `CefResourceRequestHandler`
- `CefSchemeHandlerFactory`
- Cookie/Cache/Proxy 相关路径

### 4.2 建议实践

- 自定义一个协议（如 `app://`）
- 拦截一个 URL 并返回本地内容
- 记录一次完整加载链路（request -> response -> render）

阶段产出：

- 一个可运行 demo（自定义 scheme + 拦截请求）
- 一份“网络请求生命周期笔记”

---

## 5. JS 交互与 V8（第 8 周，能力扩展）

目标：打通 C++ 与前端脚本交互。

### 5.1 重点主题

- `CefV8Handler`
- `CefMessageRouter`
- 渲染进程侧 JS 注入时机

### 5.2 建议实践

- 暴露一个 JS API 给页面调用
- 完成一次 JS -> C++ -> JS 回调闭环

阶段产出：

- 一个可复用的 JS Bridge 最小实现

---

## 6. 进入源码修改与回归（第 9-10 周，深入）

目标：可以做小改动并稳定验证，不再只读代码。

### 6.1 修改类型（先小后大）

- 日志增强（最安全）
- 行为开关（加 flag）
- 小功能修复（有明确复现）

### 6.2 修改流程建议

1. 先写问题复现步骤
2. 定位到最小改动点
3. 编译验证（最小目标）
4. 跑回归清单
5. 记录 patch 原因和影响范围

### 6.3 回归清单（最小集）

- 启动/关闭稳定性
- 页面加载与导航
- JS 交互
- 文件下载/上传
- 崩溃恢复能力

阶段产出：

- 至少 2 个小 patch + 对应验证记录

---

## 7. 每周固定模板（建议复制使用）

每周学习记录建议包含：

- 本周目标（1-3 条）
- 本周读过的文件列表
- 本周关键调用链
- 本周疑问与结论
- 下周计划

建议你每周至少输出一个“源码追踪报告”（不求长，求可复现）。

---

## 8. 推荐阅读顺序（从简单到深入）

1. `cef/include`（先 API 语义）
2. `cef/libcef`（看 CEF 如何转发到 Chromium）
3. `content/browser` + `content/renderer`（看进程职责）
4. `components`（补通用模块知识）
5. 与你业务相关模块（网络、下载、渲染、脚本等）

---

## 9. 学习成功标准（你可以自测）

达到以下 5 条，说明你已进入“源码可用阶段”：

- 能解释 3 条核心调用链（初始化、建 Browser、消息）
- 能独立定位一个回调的线程和进程上下文
- 能修改一处源码并完成最小回归
- 能说明某个行为是 CEF 层还是 Chromium 层决定
- 能把一次问题定位过程写成可复现文档

---

## 10. 给你当前状态的建议

你已经做过 Wrapper 集成和编译优化，这非常好。  
最适合你的下一步是：

- 先集中攻克“API -> 实现映射 + 多进程 IPC”两块
- 再做“网络拦截 + JS Bridge”两个实战专题
- 最后开始做小 patch（日志增强或行为开关）

这样能最快从“工程集成者”升级为“CEF 源码开发者”。
